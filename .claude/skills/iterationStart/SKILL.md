# iterationStart

Аргумент — номер итерации N (из `$ARGUMENTS`). Это **полный драйвер итерации**: спека → реализация → ревью → полировка (`/simplify`). Код добивается качества **здесь** — `/iterationClose` потом только церемония без правок кода (доки, ROADMAP, commit-месседж).

## Фазы (по порядку)

### Фаза 0 — Контекст
1. Прочитай строку #N в `ROADMAP.md`: **колонку «🧵 красная нить»** — какой резюме-ключевик итерация обязана сделать демонстрируемым, колонку «Учебный кластер» (какая ОДНА незнакомая тема вводится), existence-gate и **колонку «Питоний аналог»** — через какой знакомый инструмент осваивается новое (правило 2: концепции минимизируем, каждая обязана иметь питоний аналог). Сверься с разделами ROADMAP «Сквозные контракты» (state machine №1, error envelope №2, `openapi.json` только скриптом №3, env `DW_` №4, пагинация №5, nginx-маршрутизация №6, детерминизм AI №7) и «Заметки»: у итерации может быть заготовленное решение — локально переизобретать нельзя.
2. Прочитай `CLAUDE.md` (north-star раздел «Красная нить» + главные правила: existence-gate правило 1, одна незнакомая концепция-кластер правило 2, домен-фикстура Fernwood Supplies правило 3, AI-дисциплина правило 4, контракт только через `openapi.json` правило 5, швы правило 6, наглядность финала правила 7-8).
3. **Если плотно** (особенно тяжёлые итерации — №6 фронт-пайплайн: Vite + Orval + TanStack Query + UI + nginx + полный compose разом; №9 SSE: стрим OpenAI → Fastify-ответ → ручной клиент → UI → nginx без буферизации) — предложи расщепить и **остановись**, прежде чем писать спеку.

### Фаза 1 — Спека
4. Создай `specs/NN/spec.md` (NN двузначный) по шаблону ниже. **Тонко: ≤1 страницы.** Спека обязана **процитировать пункт красной нити** из ROADMAP как цель итерации и **назвать питоний аналог** новой концепции. Покажи её словами.

### Фаза 2 — Реализация
5. Реализуй по спеке до выполнения **done-gate** (по факту существования). Соблюдай швы (`CLAUDE.md` правило 6): composition root — дерево плагинов + фабрика `buildApp(deps)`, зависимости (Prisma-клиент, OpenAI-клиент) — аргументами и декораторами, DI-контейнеров нет; domain-логика (матрица переходов и пр.) — чистые функции без импорта Fastify/Prisma; хендлеры тонкие; env — **только** в config-модуле, префикс `DW_`. Соблюдай контракт (правило 5): `web/` никогда не импортирует TypeScript из `api/`, типы фронта — только из Orval по `api/openapi.json`, сам файл генерится **только** скриптом `openapi:emit` и руками не правится, DTO ≠ модель БД (автоген Zod-из-Prisma запрещён). Не отрезай финальную витрину (правила 7-8): выбирай реализацию, чей результат потом можно показать (кадры — ROADMAP → «Витрина»).
6. **AI-дисциплина (правило 4):** официальный `openai` SDK через декорированный клиент, снапшот пиннён в `DW_OPENAI_MODEL` (имя матчит `-\d{4}-\d{2}-\d{2}$`), `temperature=0`, промпты — константы. В тестах OpenAI **всегда мок** (фейковый клиент аргументом в `buildApp`), CI — без сети и $0. Одиночные live-вызовы — копейки (весь проект <$1), но **батч-прогоны с расходом — спроси** и назови оценку стоимости.

### Фаза 3 — Ревью и полировка (финальные шаги реализации)
Гоняется, когда done-gate функционально достигнут и код субъективно готов (не по полу-написанному). Цель — **качество кода, НЕ артефакты**: находки живут только в контексте, применяются, испаряются. Никаких `docs/reviews/`.

7. **Два ревьюера параллельно** (в одном сообщении, два вызова Agent — свежий контекст):
   - `general-reviewer` — логика/edge-cases/простота/scope-creep;
   - `constitution-reviewer` — швы `CLAUDE.md` + консистентность с аналогами.
   Каждому передай путь `specs/NN/spec.md`. Дифф они собирают сами с рабочего дерева — НЕ передавай его.
8. **Дедуп.** Одна проблема в одном месте под разными углами → склей, severity = максимум, помечай источник.
9. **Аудитор.** Вызови `review-auditor`, вставив склеенный список находок в промпт. Он выносит KEEP / DOWNGRADE→X / DROP / REWORD — страж против кривого авто-фикса по ложному срабатыванию.

   **Результат аудитора забирай pull, не push.** Push-нотификация о завершении агента иногда не доходит (наблюдалось: агент отработал за ~20с и закончил `end_turn`, но `<task-notification>` не прилетела — выглядит как «зависание», хотя вердикт готов). Поэтому: если нотификация не пришла в разумный срок (~60–90с), **не жди дальше** — вытащи финальный ответ прямо из транскрипта агента по пути `output_file`, который вернул вызов Agent. **Только хирургически через `jq`** (правило 9; не `Read`/`cat`/`tail` целиком — это переполнит контекст):
   `jq -r 'select(.message.content) | .message.content[]? | select(.type=="text") | .text' <output_file> | tail -40`
   Если у последней записи `stop_reason=end_turn` — агент завершился штатно, бери этот текст как вердикт. Если транскрипт реально оборвался на pending tool_use (а не на тексте) — вот это настоящий зависший вызов: останови агент (`TaskStop`) и применяй **консервативный фолбэк** (см. шаг 10).
10. **Применить вердикты:** DROP → выбросить; DOWNGRADE → сменить severity; REWORD → поправить якорь. Затем **CRITICAL + BUG → чини**; **NIT** → только если правка очевидно безопасна. После фиксов прогони **точечные** vitest-тесты (мок OpenAI, без сети), чтобы фикс не сломал happy-path.
    - **Консервативный фолбэк (аудитор реально недоступен).** Если вердикт не достать ни push, ни pull: выступи аудитором сам по его же критериям, но строго в защиту кода — для каждой находки по умолчанию KEEP только то, что чинишь **очевидно безопасно** (NIT-косметика), а на спорное/CRITICAL без явного подтверждения **не накладывай авто-фикс**. Лучше оставить находку на ревью пользователю, чем испортить корректный код. Отметь в хэндоффе, что аудитор был недоступен.
11. Существенные фиксы — можно прогнать ревьюеров повторно. Цель фазы: CRITICAL/BUG = 0.
12. **Причесать код.** Когда ревью чисто, вызови встроенный навык `/simplify` по текущему диффу — косметические правки качества (reuse/simplification/efficiency/altitude) поверх результата ревью, включая код самих фиксов. Это не охота на баги — корректность и швы уже отловлены ревьюерами. После — прогони **точечные** vitest-тесты, чтобы полировка не сломала happy-path.

### Фаза 4 — Хэндофф
13. Короткая сводка: done-gate выполнен; **какой пункт красной нити стал демонстрируемым**; ревью — 2-4 строки (что починил по severity, что аудитор отбросил); что причесал `/simplify`; осознанные срезы, если решили что-то НЕ чинить (пойдут в `learnings.md` при close).
    - **Итерация ещё НЕ закрыта.** Реализация и ревью — готовы, но ROADMAP **остаётся ⬜**, доки не сгенерированы, изменения не застейджены, пока не прогнан `/iterationClose N`. Явно скажи это пользователю и предложи следующий шаг — `/iterationClose N`.

## Правила
- Спека тонкая (≤1 страницы). Не раздувай — детали оркестрации ревью живут здесь, в скилле, не в спеке.
- Не предлагай батч-прогоны live OpenAI без явной пометки про стоимость (одиночные вызовы — копейки, батчи — спросить).
- Микро-решения по ходу (напр. выбор конкретного пиннёного снапшота модели; вёрстка UI-страниц; состав seed-тикетов) — спрашивай, если развилка существенна; заготовленные варианты смотри в ROADMAP → «Сквозные контракты» и «Заметки». Развилки стека (Fastify/Zod/Orval/…) уже решены — не пересматривай (CLAUDE.md → «Стек»).
