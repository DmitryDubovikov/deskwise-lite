# Итерация 00 — Каркас (Node/TS-анатомия)

> 🎯 **Тот же инженерный стандарт, другой рантайм.** Existence-gate, не accuracy-gate:
> итерация готова, когда скелет *работает и виден* — сервер отвечает, тест зелёный, линт
> чист, — а не когда он «идеален».

## Зачем это (продукт и ценность)

У продукта две аудитории, и ценность у каждой своя.

**Для команды поддержки** deskwise-lite — это мини-кабинет вымышленной компании **Fernwood
Supplies** (интернет-магазин канцтоваров): агент видит очередь клиентских тикетов, ведёт
каждый по жизненному циклу статусов и получает от AI резюме тикета и черновик ответа — вместо
ручного разбора почтового ящика. Это польза «в лоб», её щупает конечный пользователь.

**Для инженерной команды** ценность в том, *как этот кабинет устроен внутри*. Кабинет — это
фронтенд (React) плюс бэкенд (Fastify API), и их вечная беда — рассинхрон: бэк переименовал
поле тикета, фронт про это не знает и падает у пользователя в проде. deskwise-lite строится
так, что типы фронта **порождаются из той же схемы, что и бэкенд** (через `openapi.json`).
Поэтому рассинхрон физически невозможен незаметно: он ломает компиляцию фронта и краснит CI
ещё до мёрджа, а не всплывает в проде. Для *этого учебного* проекта именно вторая ценность —
главная: end-to-end type safety и есть та практика, ради строки в резюме которой проект
затеян (см. north-star в `CLAUDE.md`).

Эта, нулевая, итерация ещё ничего из продуктовой пользы не отдаёт — она кладёт **пустой
фундамент, на котором всё остальное будет стоять**. Появился работающий бэкенд-скелет:
HTTP-сервер, который поднимается одной командой и отвечает на запрос проверки живости. Без
этого шага некуда вешать ни схемы, ни домен тикетов, ни AI. Ценность здесь —
инфраструктурная: заведён каркас, на котором iter 1 начинает строить schema-first API.

## 🧵 Что это дало резюме

Прямой north-star-строки эта итерация **не заявляет** (так и записано в `ROADMAP.md`:
строка iter 0 в колонке «красная нить» — прочерк, «каркас»). Она готовит почву: поднятый
Fastify-сервер и связка pnpm + TypeScript-под-ESM + vitest + Biome — это тот рантаймовый
фундамент, поверх которого iter 1 сделает демонстрируемой первую практику (schema-first
Node API). Артефакт-доказательство самой итерации — зелёный `make check` и отвечающий
`curl http://localhost:3000/health`.

## TL;DR (простыми словами)

Было: пустой репозиторий с одной лишь конституцией и планом. Стало: внутри `api/` живёт
самостоятельный Node-пакет — Fastify-сервер с единственным роутом `GET /health`, который
поднимается через `make dev` и отвечает `{"status":"ok"}`. К нему прикручены три инструмента
качества: vitest гоняет смок-тест роута без поднятия сети, Biome проверяет линт и формат,
TypeScript-компилятор в строгом режиме проверяет типы. Всё это дёргается одной целью
`make check`. Добавили, по сути, два кусочка: (1) работающий HTTP-скелет и (2) обвязку
проверок вокруг него.

## Что это за техника (мини-ликбез через питоний аналог)

Ни одна из вещей ниже не нова *по идее* — все они имеют прямой аналог в Python-опыте автора.
Ново только **оборудование** (сам рантайм). Поэтому осваиваем анатомию Node/TS-проекта как
«тот же скелет, другие кости».

- **package.json + pnpm** ≈ `pyproject.toml` + `uv`. `package.json` — манифест пакета:
  имя, зависимости, скрипты (`dev`/`test`/`check`). `pnpm` — менеджер пакетов, который эти
  зависимости ставит и пишет точный `pnpm-lock.yaml` (аналог `uv.lock`) — лок-файл фиксирует
  *точные* версии, чтобы установка была воспроизводима. Ключевое семейное решение:
  **никаких workspaces** — `api/` (а позже `web/`) остаются независимыми пакетами со своими
  локами (правило 5 конституции).

- **tsconfig.json под ESM/NodeNext** ≈ настройки интерпретатора и `mypy` в одном. Это конфиг
  TypeScript-компилятора. Два незнакомых параметра: `module`/`moduleResolution: "NodeNext"`
  говорят «мы пишем современные ES-модули (`import`/`export`), а не старый CommonJS
  (`require`)» — примерно как разница «пакеты с `__init__.py` против голых скриптов»; а
  `strict: true` включает жёсткую проверку типов (аналог строгого `mypy`), чтобы ошибки
  ловились компилятором, а не в рантайме.

- **tsx-watch** ≈ `uvicorn --reload`. `tsx` — запускалка, которая исполняет TypeScript
  напрямую, без отдельного шага компиляции, а `tsx watch` вдобавок перезапускает процесс при
  правке файла. Это dev-режим: правишь код — сервер сам перезагрузился.

- **vitest + `app.inject()`** ≈ `pytest` + `TestClient`. `vitest` — тест-раннер (как
  pytest). `app.inject({method, url})` — способ Fastify «позвать» роут **в памяти, без
  реального сетевого сокета**: строится fake-запрос, прогоняется через всю цепочку сервера,
  возвращается ответ. Ровно как `TestClient` у FastAPI гоняет запрос без поднятия uvicorn —
  тест быстрый и не занимает порт.

- **Biome** ≈ `ruff`. Один инструмент, закрывающий сразу линт (поиск подозрительного кода)
  и форматирование (единый стиль). Семья намеренно берёт один инструмент вместо связки
  ESLint+Prettier — меньше конфигов, меньше движущихся частей.

## Поток данных

Никаких данных итерация ещё не переносит и не валидирует (БД, схем, AI пока нет). «Поток»
здесь — это две команды разработчика и что каждая приводит в движение. Всё начинается с того,
что **разработчик набирает `make dev` или `make check`** — эти цели корневого `Makefile`
просто заходят в `api/` и дёргают соответствующий pnpm-скрипт.

```
make dev ──▶ (Makefile) cd api && pnpm dev ──▶ tsx watch src/index.ts
                                                      │
                                                      ▼
                                          index.ts зовёт buildServer()  (server.ts)
                                                      │  создаёт Fastify(), вешает GET /health
                                                      ▼
                                          app.listen(:3000)  ──▶  живой HTTP-сервер
                                                      │
   разработчик: curl localhost:3000/health ──────────┘  ◀── {"status":"ok"}


make check ─▶ (Makefile) cd api && pnpm check ─▶ biome check .   (линт+формат)
                                              ├─▶ tsc --noEmit   (типы, strict)
                                              └─▶ vitest run  ─▶  server.test.ts
                                                                   │ buildServer()
                                                                   ▼ app.inject(GET /health)
                                                                 assert 200 & {"status":"ok"}
```

Обрати внимание на развилку: **`make dev` поднимает настоящий сетевой сервер** (его щупаем
`curl`'ом руками), а **`make check` — нет**. Тест зовёт ту же фабрику `buildServer()`, но
прогоняет запрос через `app.inject()` в памяти, без сокета и без порта. Это одна и та же
сборка сервера, дёрнутая двумя способами: живьём для человека и in-memory для теста.

| Инструмент | Что делает | Куда пишет / что отдаёт |
|---|---|---|
| `make dev` → `pnpm dev` → `tsx watch` | компилирует TS на лету и держит живой сервер, перезапуская при правках | процесс слушает `:3000`, отвечает на HTTP |
| `curl :3000/health` | ручная проверка живого роута | печатает тело ответа `{"status":"ok"}` |
| `make check` → `biome check` | линт + проверка формата | exit code (0 = чисто) |
| `make check` → `tsc --noEmit` | проверка типов в strict, без генерации `.js` | exit code (0 = типы сходятся) |
| `make check` → `vitest run` | гоняет `server.test.ts` через `app.inject()` | зелёный/красный прогон, **сеть не нужна** |

Честные оговорки — чего в этой итерации **нет** (типичная путаница «а где же…»): нет ни
одной Zod-схемы и нет `openapi.json` (это iter 1), нет плагинов и фабрики `buildApp(deps)`
(iter 2), нет pino-логов (iter 3), нет БД/Prisma (iter 4), нет фронта, nginx и compose
(iter 6). `app.log.info` в `index.ts` при дефолтном Fastify — тихий no-op: настоящий
логгер приедет в iter 3. Здесь — только голый скелет.

## Карта «где в коде»

> Номера строк — ориентир на момент закрытия iter 0; при дрейфе ищи по именам символов
> (`buildServer`, файлы `server.ts`/`index.ts`).

1. **Фабрика сервера** — `api/src/server.ts:3`, функция `buildServer()`. Она создаёт
   инстанс Fastify, вешает на него единственный роут `GET /health`, возвращающий объект
   `{ status: "ok" }`, и отдаёт собранное приложение наружу — **не поднимая при этом
   сеть**. Именно поэтому её может звать и боевой запуск, и тест: сборка приложения
   отделена от его прослушивания порта. (Это зародыш будущей `buildApp(deps)` из iter 2 —
   пока без плагинов и зависимостей.)

   ```ts
   import Fastify from "fastify";

   export function buildServer() {
       const app = Fastify();
       app.get("/health", async () => ({ status: "ok" }));
       return app;
   }
   ```

2. **Точка входа (боевой запуск)** — `api/src/index.ts:3`. Берёт `buildServer()` и уже
   *поднимает* его на порту 3000, слушая на `0.0.0.0` (все интерфейсы — важно для будущего
   контейнера). Если старт не удался — логирует ошибку и выходит с ненулевым кодом. Разнос
   «собрать сервер» (`server.ts`) и «слушать порт» (`index.ts`) — то, что делает тест
   возможным без сокета.

   ```ts
   import { buildServer } from "./server.js";

   const app = buildServer();
   app.listen({ port: 3000, host: "0.0.0.0" }, (err, address) => {
       if (err) { app.log.error(err); process.exit(1); }
       app.log.info(`listening on ${address}`);
   });
   ```

   > Заметь `./server.js` в импорте, хотя файл называется `server.ts`. Это требование
   > ESM/NodeNext: в import указывается имя *итогового* модуля (`.js`), даже когда пишешь на
   > TypeScript. Первая «Node ≠ Python» грабля — см. `learnings.md`.

3. **Смок-тест роута** — `api/src/server.test.ts:5`. Собирает приложение той же фабрикой и
   через `app.inject()` шлёт in-memory-запрос `GET /health`, после чего проверяет и код
   ответа (200), и тело (`{ status: "ok" }`). Тест не открывает сокет и не ходит в сеть —
   аналог `pytest` + `TestClient`.

   ```ts
   const app = buildServer();
   const response = await app.inject({ method: "GET", url: "/health" });
   expect(response.statusCode).toBe(200);
   expect(response.json()).toEqual({ status: "ok" });
   ```

4. **Скрипты пакета** — `api/package.json`. Здесь заданы четыре команды, которыми оперирует
   весь проект: `dev` (`tsx watch src/index.ts`), `test` (`vitest run`), а также `lint`
   (`biome check .`) и `typecheck` (`tsc --noEmit`), собранные в агрегат `check`
   (`pnpm lint && pnpm typecheck && pnpm test`). Корневой `Makefile` в них только
   проксирует.

   ```json
   "scripts": {
       "dev": "tsx watch src/index.ts",
       "test": "vitest run",
       "lint": "biome check .",
       "typecheck": "tsc --noEmit",
       "check": "pnpm lint && pnpm typecheck && pnpm test"
   }
   ```

5. **Корневой Makefile** — `Makefile`. Три цели (`dev`/`test`/`check`), каждая из которых
   заходит в `api/` и дёргает одноимённый pnpm-скрипт. Это единая «входная дверь» проекта из
   корня — знакомое семейное оборудование, вводится бесплатно.

   ```makefile
   dev:
   	cd api && pnpm dev
   check:
   	cd api && pnpm check
   ```
