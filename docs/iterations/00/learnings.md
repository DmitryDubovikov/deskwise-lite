# Learnings — Итерация 00 (каркас)

## Принятые решения и почему

- **`buildServer()` отделён от `app.listen()`** (`server.ts` vs `index.ts`). Фабрика собирает
  приложение, точка входа его слушает. Сделано так с самого начала, потому что это то, что
  позволяет тесту звать `app.inject()` без сокета — и это же зародыш будущей
  `buildApp(deps)` из iter 2. Дешевле заложить шов сразу, чем потом расщеплять монолитный
  `index.ts`.

- **`host: "0.0.0.0"` в `listen`**, а не `localhost`. На будущее: внутри Docker-контейнера
  сервер, слушающий только `127.0.0.1`, недостижим снаружи. Контейнера ещё нет (iter 4/6),
  но правильный хост стоит копейки сейчас и снимает будущую грабельку.

- **Node/TS-концепции держим минимальными** (правило 2): один пакет, один роут, один тест.
  Объём кода крошечный — вся нагрузка итерации в *незнакомом оборудовании* (ESM/tsconfig,
  pnpm, tsx, vitest, Biome), а не в объёме.

## Грабли: где Node повёл себя не как Python

- **Импорт с расширением `.js`, хотя файл `.ts`.** `index.ts` импортирует
  `"./server.js"`, `server.test.ts` — тоже `"./server.js"`, при том что на диске лежит
  `server.ts`. Это требование ESM/NodeNext: в спецификаторе указывается имя *итогового*
  модуля, а компилятор сам сопоставляет его с `.ts`-исходником. В Python `from . import
  server` не знает ни о каких расширениях — здесь же пропуск `.js` даёт ошибку резолвинга
  модуля. Первый «Node ≠ Python» сюрприз; принять как данность NodeNext-режима.

- **Дефолтный Fastify-логгер молчит.** `app.log.info("listening on…")` в `index.ts` при
  `Fastify()` без опции `logger` — тихий no-op, в консоль ничего не печатается. В Python
  привычка «залогировал — увидел» здесь не работает: логгер надо явно включить (и это тема
  iter 3 — pino). Поэтому смок сервера в `demo.md` доказывается ответом `curl`'а, а не
  строкой в логе.

- **pnpm по умолчанию блокирует build-скрипты зависимостей.** `tsx` тянет `esbuild`, у
  которого есть `postinstall`-скрипт (`node install.js`, докачивает нативный бинарь). pnpm
  v10+ такие скрипты по умолчанию **не запускает** ради безопасности и требует явного
  одобрения. Одобрение живёт в `api/pnpm-workspace.yaml` (ключ `allowBuilds`) — без него
  свежий `pnpm install` оставил бы esbuild недособранным и `tsx` бы не запустился. Проверено
  на чистой установке (`rm -rf node_modules && pnpm install`): esbuild-postinstall
  отработал, предупреждения об «ignored build scripts» нет, `make check` зелёный. В Python с
  `uv`/`pip` такого барьера на build-скрипты по умолчанию нет — ещё одно расхождение.

## Сознательное решение (по итогам осмотра): `pnpm-workspace.yaml` ≠ workspaces

Конституция (правило 5, «Что осознанно НЕ делаем») запрещает **pnpm workspaces**. В `api/`
при этом лежит файл `pnpm-workspace.yaml` — это может смутить будущего-меня. Фиксирую, что
**противоречия нет**: файл содержит только `allowBuilds` (одобрение build-скрипта esbuild,
см. грабли выше) и **не объявляет `packages:`** — то есть не создаёт мульти-пакетного
workspace-of-packages. В pnpm v10+ этот файл — штатное место для конфигурации одобрения
build-скриптов даже у одиночного пакета. `api/` остаётся самостоятельным пакетом со своим
`pnpm-lock.yaml`. Запрет из правила 5 — на *workspace как способ связать несколько пакетов
общими типами* (чего мы избегаем ради контракта через `openapi.json`), а не на сам файл.
Осознанно **не удаляем** файл: без него ломается сборка esbuild. Демо-шаг 5 в `demo.md`
проверяет ровно это — отсутствие объявленных членов workspace.
