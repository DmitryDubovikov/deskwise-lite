# Learnings 05 — REST-сборка

## Решения и почему

- **`z.strictObject` для PATCH-DTO вместо дефолтного `z.object`.** Обычный `z.object`
  молча стрипает неизвестные ключи — клиент, пославший `{"status":"closed"}` в PATCH,
  получил бы 200 и уверенность, что статус изменился. Strict-вариант превращает это в
  явный 400 — state machine (контракт №1) нельзя обойти даже по незнанию. Это
  единственное «микро-новое» итерации; остальное — сборка выученного.

- **Enum статусов рождается в domain, Zod его импортирует.** `TICKET_STATUSES` объявлен
  в `src/domain/ticket-status.ts`, а `schemas/ticket.ts` делает `z.enum(TICKET_STATUSES)`.
  Направление зависимости выбрано осознанно (схема зависит от домена, не наоборот):
  доменная матрица остаётся без единого импорта инфраструктуры, и её unit-тест гоняет
  все 16 пар статусов без БД.

- **PATCH/DELETE — атомарной мутацией с ловлей `P2025`, а не `findUnique` + мутация.**
  Один round-trip к БД вместо двух и отсутствие гонки «проверил — а запись уже удалили».
  Prisma кидает типизированную `PrismaClientKnownRequestError` с кодом `P2025` — предикат
  `isRecordNotFound` в `routes/tickets.ts` превращает её в канонный 404. Питоний рефлекс
  «сначала `get_object_or_404`» здесь дороже идиомы «сделай и поймай».

- **Transition — наоборот, через `findUnique` + `update`.** Матрице нужен текущий статус
  *до* мутации, поэтому паре запросов тут есть причина. Между чтением и записью есть
  теоретическая гонка (два конкурентных перехода одного тикета оба увидят `open`);
  для фикстуры с одним пользователем это осознанно принято —
  `# dw-lite: findUnique+update → атомарный updateMany({where: {id, status: from}})`,
  апгрейд известен и не делается.

- **Seed идемпотентен через фиксированные UUID + upsert, а не `createMany`.**
  `createMany` на втором прогоне либо падает, либо (со `skipDuplicates`) не возвращает
  канон после ручных правок. Upsert с фиксированными id (`00000000-0000-4000-8000-…N`)
  делает повторный прогон «сбросом к канону» — витрина и AI-демо всегда воспроизводимы.
  Все 30 upsert'ов завёрнуты в один `$transaction` — батч-round-trip плюс атомарность.

- **Фабрика `createPrismaClient` вынесена в `src/db.ts`.** Потребителей стало три
  (app-fallback, test-setup, seed), и все обязаны собирать клиент одинаково (адаптер
  `PrismaPg`, URL из `loadConfig`). Правило 6 не размылось: env по-прежнему читается
  только в config-модуле, фабрика лишь переиспользует его.

## Грабли (где Node повёл себя не как Python)

- **204 и `z.null()`: FastAPI просто молчит, fastify-type-provider-zod — нет.**
  В FastAPI `status_code=204` даёт ответ без тела и точка. Здесь `response: {204: z.null()}`
  сериализуется в спеку как `content: {"application/json": {schema: {enum: [null]}}}` —
  контракт обещает JSON-тело, которого в HTTP-ответе нет, и Orval сгенерил бы клиенту
  парсинг пустоты. Лечится хуком `transformObject` в `app.ts`, вычищающим `content` у
  всех 204 — в одном месте для обоих потребителей спеки (`openapi:emit` и `/docs`).

- **У Prisma нет встроенных фикстур — seed это просто скрипт по конвенции.** В Django
  `loaddata` штатный; здесь пришлось зарегистрировать раннер в `prisma.config.ts`
  (`migrations.seed: "tsx prisma/seed.ts"`), а идемпотентность — полностью твоя забота
  (см. решение про upsert).

- **Типы seed-данных — из generated-клиента, не из domain.** Seed пишет в БД, поэтому
  `TicketStatus`/`TicketPriority` импортируются из `src/generated/prisma/enums.js` (слой
  хранения), а не из `src/domain/ticket-status.ts`. Значения сейчас совпадают, но слои
  разные — DTO ≠ модель БД (правило 5), и seed принадлежит стороне хранения.

## Сознательные срезы и отложенное

- `# dw-lite: orderBy id → orderBy createdAt` — у тикета нет `createdAt`: домен заморожен
  (правило 3), поле не добавляем; стабильность страниц даёт `orderBy id`.
- `# dw-lite: offset → cursor` — зафиксированный не-апгрейд пагинации (контракт №5).
- `# dw-lite: envelope → RFC 9457` — зафиксированный не-апгрейд формата ошибок (№2).
- Гонка в transition (см. решения) — принята для фикстуры.
- **Домен с этого момента заморожен:** новых сущностей и полей не будет, масштаб — числом
  тикетов в seed. Следующая работа — вокруг домена: фронт (iter 6), CI-гейт (iter 7),
  AI-эндпоинты (iter 8–9).
