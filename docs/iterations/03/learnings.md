# Learnings 03 — Эксплуатационная поверхность: pino, setErrorHandler, `/docs`

## Решения и почему

- **Логгер едет в `buildApp` как зависимость, а не включается внутри фабрики.** Поле
  `logger` в `AppDeps` с дефолтом `false` даёт три режима одной ручкой: прод передаёт
  `{ level }` из config-модуля, обычные тесты молчат, а тест структурных логов подсовывает
  pino in-memory stream и разбирает строки. Альтернатива — читать `DW_LOG_LEVEL` внутри
  `buildApp` — нарушила бы контракт №4 (env читается только в config-модуле) и заставила
  бы тесты патчить окружение.

- **Словарь машинных кодов — TypeScript-union, а не Zod-enum в спеке.** В
  `ErrorResponseSchema` поле `code` — просто `z.string()`; союз `ErrorCode` живёт рядом и
  проверяет только серверный код (`errorBody` не даст опечататься). Захоти мы
  `z.enum([...])` — каждый новый код (409 `CONFLICT` в iter 5) менял бы JSON Schema всех
  ошибочных ответов во всех роутах, то есть ломал бы контракт по всей спеке ради
  расширения словаря. Цена решения: клиент не увидит перечень кодов в спеке — приемлемо
  для «запаски», словарь задокументирован комментарием в `schemas/error.ts`.

- **Транспортные коды Fastify (`FST_ERR_CTP_*`) замаппены в `VALIDATION_ERROR`.** Битый
  JSON и кривой content-type — с точки зрения клиента та же «ты прислал невалидный
  запрос», что и ошибка Zod. Выносить внутренние имена фреймворка в контрактный словарь —
  значит навсегда привязать клиентов к деталям Fastify. Остальные не-500 ошибки
  фреймворка получают запасной код `REQUEST_ERROR`, чтобы ветка «прочее 4xx» не
  прикидывалась ни валидацией, ни 500-й.

- **`errorResponses` — спред общих кодов, специфичные — точечно.** Объект
  `{400, 500}` подмешивается в response-мапу каждого роута (это коды, которые может
  вернуть любой роут с валидируемым входом), а 404 объявляет только `GET /tickets/:id`.
  Так спека не врёт «этот роут умеет 404», когда хендлеру нечего терять.

- **500 отвечает generic-сообщением, детали — только в лог.** `setErrorHandler` на
  ветке 500 зовёт `request.log.error({ err })` и отдаёт клиенту константный message.
  Тест держит это равенством всего тела ответа — стоит кому-то «улучшить» хендлер до
  проброса `error.message`, тест покраснеет строкой `secret internal detail`.

## Грабли («где Node повёл себя не как Python»)

- **404 незнакомого роута не проходит через error handler.** В FastAPI и валидация, и
  «нет такого пути» — исключения, ловящиеся `@app.exception_handler`. В Fastify маршрут,
  которого нет, — не ошибка, а отдельный сценарий со своим `setNotFoundHandler`; без него
  envelope покрывал бы всё, кроме самого частого случая. Единый формат ошибок в Fastify —
  это **два** хендлера, не один.

- **Ошибок «плохой запрос» два сорта, и Zod-хелпер видит только один.** Ошибка
  Zod-валидации распознаётся хелпером `hasZodFastifySchemaValidationErrors` из
  `fastify-type-provider-zod`, но битый JSON падает **раньше** валидации — в content-type
  парсере Fastify, с кодом `FST_ERR_CTP_*` и статусом 400. Наивный error handler «Zod →
  400, остальное → 500» превращал бы опечатку в JSON клиента в Internal Server Error.
  Отсюда отдельная ветка по префиксу `error.code`.

- **pino не ставится — он уже внутри.** Ожидание из Python-мира «поставь structlog,
  настрой процессоры, обвяжи middleware контекстом запроса» здесь схлопывается в опцию
  `logger` конструктора Fastify: JSON-формат, уровни и `reqId` в каждой запросной строке —
  из коробки. Package.json итерации не получил ни одной новой logging-зависимости
  (единственная новая — `@fastify/swagger-ui`).

- **`z.coerce` обязателен для чисел из env.** `process.env` в Node — всегда строки;
  `z.number()` на `DW_PORT` падал бы на валидном `.env`. Pydantic коэрсит молча, Zod
  требует сказать это явно — `z.coerce.number().int().positive()`.

## Сознательные срезы (`dw-lite:`-маркеры и решения)

- `schemas/error.ts:4` — **envelope вместо RFC 9457 problem+json**:
  `// dw-lite: envelope → RFC 9457 problem+json — зафиксированный не-апгрейд` (контракт №2).
- `routes/tickets.ts:40` — **голый массив** в `GET /tickets` остаётся до iter 5:
  `// dw-lite: голый массив → {items, total, page, limit} (контракт №5, iter 5)`.
- **Логи — только stdout.** pino-pretty, транспорты, ротация, метрики/трейсинг — вне
  scope по спеке; для витрины JSON-строки из терминала достаточно.
- **`DW_LOG_LEVEL` не энумерируется** — `z.string()` без списка уровней: опечатка в
  уровне уронит процесс на старте понятной ошибкой pino, отдельная валидация в конфиге
  дублировала бы её ради красоты сообщения.
