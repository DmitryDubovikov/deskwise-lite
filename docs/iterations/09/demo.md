# Демо 09 — SSE token streaming: токены летят из OpenAI в терминал и UI

Прогон доказывает красную нить итерации: стрим токенов OpenAI протекает через Fastify
SSE-ответом и доезжает до клиента порциями, без буферизации по всему пути (api → nginx →
браузер). Зачем это за пределами демо: стриминг — витринная сила Node (async I/O), и
именно он делает AI-фичу пригодной для человека — агент видит первые слова черновика
сразу, а не спиннер до конца генерации. Заодно проверяем границу контракт-подхода:
SSE-эндпоинт осознанно вне `openapi.json`, и контракт от итерации не должен был
шелохнуться.

Все команды запускаются из корня репо: `/Users/dd/projects/pet/deskwise-lite`. Стек:
`make up` (nginx :8080 + api + db), тикеты засеяны `make seed`. Для шагов 1–3 сеть/ключ
не нужны; шаг 4 — один live-вызов (⚠️ копейки), ему нужен реальный `DW_OPENAI_API_KEY`
в `api/.env` (одной строкой: `echo 'DW_OPENAI_API_KEY=<твой ключ с platform.openai.com>'
>> api/.env` — единственное значение, которое знает только владелец ключа).

## 1. Контракт не шелохнулся: роут скрыт из спеки, регенерация идемпотентна

Граница контракт-подхода — явная: suggest-reply в `openapi.json` отсутствует (`hide`), а
`make generate` обязан дать то же содержимое побайтно — иначе итерация незаметно
подвинула контракт. Проверка хеш-снапшотом (работает и на незакоммиченном дереве), двойной
прогон заодно доказывает идемпотентность регенерации.

```bash
jq '.paths | keys' api/openapi.json
snap() { { shasum -a 256 api/openapi.json; find web/src/generated -type f | sort | xargs shasum -a 256; } | shasum -a 256; }
S0=$(snap); make generate >/dev/null 2>&1; S1=$(snap); make generate >/dev/null 2>&1; S2=$(snap)
[ "$S0" = "$S1" ] && [ "$S1" = "$S2" ] && echo CONTRACT-STABLE || echo CONTRACT-DRIFT
```

**Ожидаемо:** в списке путей есть `/tickets/{id}/summarize`, но **нет**
`/tickets/{id}/suggest-reply`; финальная строка — `CONTRACT-STABLE`.

## 2. Тесты с фейковым стримом — wire-формат целиком, без сети

Доказываем контракт №7 и сам wire-формат: фейковый стрим через шов `buildApp(deps)` выдаёт
заготовленные дельты, а тест ассертит тело SSE-ответа побайтно — порядок кадров,
терминатор `[DONE]`, кадр `event: error` при обрыве, 404 без вызова клиента.

```bash
cd api && pnpm vitest run suggest-reply; cd ..
```

**Ожидаемо:** `Test Files 1 passed`, `Tests 3 passed` — стрим кадров по порядку +
`[DONE]`; обрыв посреди стрима → `event: error` с envelope
`{"error":{"code":"STREAM_ERROR",…}}` и без `[DONE]`; несуществующий тикет → 404, клиент
не позван. Сети нет: нужен только Postgres из compose. Побочный эффект: тесты чистят
таблицу тикетов в dev-Postgres — шаг 4 начинается с повторного `make seed`.

## 3. Ошибочный путь через продуктовую поверхность: 404 до старта стрима — обычный JSON

До первого SSE-кадра заголовки ещё не ушли, поэтому несуществующий тикет обязан дать
обычный 404 в едином envelope — как у всех эндпоинтов, никакого SSE. Проверяем через
nginx, ключ не нужен (до OpenAI дело не доходит).

```bash
curl -si -X POST http://localhost:8080/api/tickets/00000000-0000-0000-0000-000000000000/suggest-reply | head -4
curl -s -X POST http://localhost:8080/api/tickets/00000000-0000-0000-0000-000000000000/suggest-reply | jq .
```

**Ожидаемо:** статус `HTTP/1.1 404 Not Found`, `content-type: application/json` (не
`text/event-stream`), тело — единый envelope:

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Ticket not found"
  }
}
```

## 4. Токены летят через nginx порциями ⚠️ live, стоит денег (один вызов — копейки)

Кульминация и правило 8: сам артефакт стриминга — сырые SSE-кадры, приходящие в терминал
**в разное время**. `curl -N` выключает буферизацию curl'а, а perl-однострочник штампует
миллисекундное время прихода каждой строки (`date +%N` на macOS не работает) — если бы
nginx буферизовал (или буферизовал Node), все кадры имели бы одну метку. Путь — через
nginx :8080, то есть ровно тот, которым ходит браузер: это заодно доказывает SSE-локацию
с `proxy_buffering off`.

```bash
make seed
TICKET_ID=$(curl -s "http://localhost:8080/api/tickets?limit=1" | jq -r '.items[0].id')
curl -sN -X POST "http://localhost:8080/api/tickets/$TICKET_ID/suggest-reply" \
  | perl -MTime::HiRes=time -ne 'BEGIN{$|=1} printf "%.3f  %s", time, $_ if /\S/'
```

**Ожидаемо:** десятки строк вида `data: {"delta":"…"}` с **разными** метками времени —
между кадрами единицы-десятки миллисекунд, дельты приходят по мере генерации, а не одной
меткой одним куском в конце; последняя строка — `data: [DONE]`. Содержимое дельт, склеенное подряд, — короткий
вежливый английский черновик ответа именно на этот тикет. Повторный прогон стрима
состояние не мутирует (черновик нигде не сохраняется — в БД ничего не изменилось). Батч
по всем тикетам НЕ гоняем (правило 4).

## 5. UI: черновик дорисовывается на глазах (кадр витрины, глазами)

Финальный продуктовый вид — для витрины, не для доказательства (правило 8): открой
`http://localhost:8080`, выбери тикет, нажми **Suggest reply**.

**Ожидаемо:** кнопка меняется на «Streaming…», под телом тикета появляется зелёная
карточка черновика, и текст в ней дописывается порциями по мере генерации. Переключение
на другой тикет сбрасывает черновик (ремоунт по `key={selectedId}`).
