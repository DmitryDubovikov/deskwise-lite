# Learnings 06 — фронт-пайплайн

## Решения и почему

- **Одна итерация фазами A/B, а не 6a/6b (решение 2026-07-23).** Контракт-пайплайн и UI
  сцеплены одним диффом (сгенерённый клиент бессмысленно ревьюить без потребителя),
  поэтому итерацию не расщепляли: реализация шла фазами, а ревью-пайплайн прошёл один раз
  по всему диффу. Налог церемонии — на итерацию, дробление его бы удвоило.
- **`.meta({id})` вместо инлайновых схем.** Без имён в `components/schemas` Orval честно
  генерит типы вида `listTickets200ItemsItem` — работает, но нечитаемо и не переживает
  перестановку полей. `.meta({id: "Ticket"})` на Zod-схеме + `jsonSchemaTransformObject`
  в `transformObject` дают `$ref` и человеческие имена. Это метаданные контракта, не смена
  shape — заморозка домена (правило 3) не нарушена.
- **Сгенерённое коммитится.** И `api/openapi.json`, и `web/src/generated/` живут в git —
  осознанно: это вход contract-drift-gate iter 7 (`git diff --exit-code` после
  регенерации), и ревьюеру виден фактический контракт, которым живёт фронт.
- **`ApiError implements ErrorResponse`.** Мутатор приводит любой отказ (не-2xx и даже
  сетевой `TypeError` из `fetch`) к исключению формы envelope №2; `implements` с type-only
  импортом из `generated/` заставляет компилятор следить, что рантайм-исключение совпадает
  с контрактным типом ошибки, — компоненты читают `error.error.message` без проверок, и
  рантайм-цикла с генерацией нет (импорт стирается при сборке).
- **Матрица переходов живёт только в домене api.** UI рендерит кнопки на все статусы,
  кроме текущего, и показывает 409 из envelope как есть. Дублирование матрицы на фронте
  дало бы «умные» кнопки ценой второго источника истины — ровно тот рассинхрон, против
  которого весь проект.
- **`tsc --noEmit` — часть `pnpm build` и Docker-сборки web.** Рассинхрон с контрактом
  валит не только локальный `pnpm typecheck`, но и `docker compose build` — герой-свойство
  дотянуто до оркестрации.
- **Vite dev proxy повторяет прод-nginx.** Одно и то же правило (`/api/*` → апстрим со
  срезом префикса) в `vite.config.ts` и `nginx.conf` — `customFetch` не знает, где живёт
  API, и работает одинаково в dev и за nginx (контракт №6).

## Грабли (где Node повёл себя не как Python)

- **pnpm блокирует postinstall-скрипты зависимостей по умолчанию.** В Python `pip install`
  просто ставит пакет, а pnpm ≥10 отказывается запускать build-скрипты транзитивных
  зависимостей, пока их явно не разрешить. Пришлось завести `web/pnpm-workspace.yaml` с
  `allowBuilds`: `esbuild: true` (нужен Vite), `@scarf/scarf: false` (телеметрия из
  транзитивных зависимостей Orval — скрипт не нужен). Тот же файл заодно служит маркером
  корня пакета: pnpm не уходит искать workspace вверх по дереву (решение №3 — без
  workspaces).
- **corepack и `packageManager`.** Чтобы Docker-образы собирались тем же pnpm, что и
  локально, в оба `package.json` добавлено поле `packageManager: "pnpm@11.13.1"`, а в
  Dockerfile — `corepack enable` (+ `COREPACK_ENABLE_DOWNLOAD_PROMPT=0`, иначе сборка
  интерактивно спрашивает подтверждение скачивания). Питоний аналог — пин версии uv, но в
  Node это встроено в сам рантайм-образ.
- **Слои Docker и `prisma generate`.** `postinstall` Prisma требует `schema.prisma` уже на
  слое `pnpm install` — поэтому в `api/Dockerfile` схема копируется вместе с lockfile, а
  каталог миграций — позже: новая миграция не инвалидирует слой зависимостей. Сгенерённый
  клиент (`src/generated/prisma`) выживает при `COPY src ./src`, потому что COPY мержит
  поверх, а в контексте сборки его нет (`.dockerignore`).
- **Provenance-аттестации BuildKit ломали идемпотентность `make up`.** Дефолтная
  аттестация встраивает timestamp сборки, поэтому даже полностью кэшированный
  `docker compose up -d --build` выдавал новый image ID — и compose пересоздавал api/web
  на каждом прогоне вместо no-op. `provenance: false` в build-секции compose (v5.1.4)
  молча игнорируется; работает переменная `BUILDX_NO_DEFAULT_ATTESTATIONS=1`, которую
  теперь ставит цель `make up`. Поймано церемонией закрытия на проверке «повторный up —
  no-op».
- **Смок-откат герой-демо — обратным perl'ом, не `git checkout`.** До коммита итерации
  `git checkout -- api/src/schemas/ticket.ts` откатил бы файл к состоянию прошлой
  итерации и уничтожил бы незакоммиченную работу. В `demo.md` откат записан симметричной
  perl-заменой — она безопасна в любом состоянии дерева.
- **Псевдо-тело у 204 (хвост из iter 5) остался актуален и для Orval:** без зачистки
  `{enum: [null]}`-контента в `transformObject` сгенерённый клиент ждал бы JSON-тело у
  DELETE-ответа. Теперь `transformObject` делает два шага: `jsonSchemaTransformObject`
  (components/$ref) + зачистка 204.

## Сознательные срезы и отложенное

- `# dw-lite: web без тестов → vitest+MSW` (`web/vite.config.ts`) — фронт без юнит-тестов:
  типы держит tsc, механику — смок демо. Апгрейд — vitest + MSW поверх того же контракта.
- `# dw-lite: tsx в контейнере → tsc build` (`api/Dockerfile`) — api в проде-фикстуре
  бежит на tsx без компиляции; для витрины это честно, компиляция не окупается.
- `# dw-lite: master-detail на useState → react-router` (`web/src/App.tsx`) — навигация
  без роутера: выбранный тикет — просто state, URL не меняется.
- SSE-локация nginx (`proxy_buffering off`) — не делалась, требование iter 9 к этому
  конфигу (отмечено комментарием в `web/nginx.conf`).
- CI drift-gate — iter 7: сегодня «пустой diff после `make generate`» проверяется руками
  (шаг 3 демо), краснеть будет CI.
