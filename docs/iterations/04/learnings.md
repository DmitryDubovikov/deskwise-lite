# Learnings 04 — Prisma-цикл

## Решения и почему

- **Prisma 7: driver adapter вместо встроенного движка.** Клиент создаётся с адаптером
  `@prisma/adapter-pg` (`new PrismaClient({ adapter: new PrismaPg({ connectionString }) })`)
  — запросы идут через обычный JS-драйвер pg, без отдельного бинарного query engine.
  Это штатный путь Prisma 7, и он же удобен нам архитектурно: строка подключения
  передаётся явно из `loadConfig()`, а не выуживается магией из env глубоко внутри
  библиотеки — правило 6 соблюдается само собой.

- **Генератор `prisma-client` с output в `src/generated/prisma`.** Клиент генерится как
  исходники прямо в дерево `src/` (а не в потроха `node_modules`), но в git не попадает —
  это артефакт сборки, его пересоздаёт хук `postinstall: prisma generate` при каждом
  `pnpm install`. Свежий клон репо получает рабочий типизированный клиент без ручных шагов.

- **`prisma.config.ts` — второе (и последнее) место чтения env.** Prisma CLI
  (migrate/generate) работает вне рантайма приложения и через `buildApp` не проходит,
  поэтому URL базы для него живёт в отдельном конфиге (блок `datasource` в самой схеме
  остался без `url`). Зафиксировано как задокументированное исключение из правила 6 —
  прямо в комментарии файла.

- **Ленивый дефолт `deps.prisma` в `buildApp`.** Без явно переданного клиента фабрика
  строит его сама из `loadConfig().databaseUrl` — благодаря этому `openapi:emit` собирает
  приложение и пишет спеку без поднятой БД (клиент создаётся, но соединение не
  открывается, пока нет запросов), а `src/index.ts` итерация вообще не тронула.

- **Один pg-пул на тестовый прогон, не на `buildApp()`.** Общий `PrismaClient` живёт в
  `test-setup.ts`, тесты передают его явно как `deps.prisma`. Наивный вариант «каждый
  тест собирает app со своим клиентом» открывал бы по пулу соединений на каждый вызов
  фабрики. Плата: файлы тестов делят одну таблицу, поэтому `fileParallelism: false` в
  `vitest.config.ts` — при нашем объёме тестов (16 штук, ~700 мс) это ничего не стоит.

- **DTO ≠ модель БД — продублировано руками, и это дизайн.** `TicketSchema` (Zod) и
  модель `Ticket` в `schema.prisma` описывают одни и те же поля независимо; дефолты
  `open`/`normal` тоже живут на обоих слоях. Правило 5 запрещает автогенерить одно из
  другого — контракт API и схема хранения обязаны уметь расходиться (в iter 5 разойдутся:
  ответ списка станет `{items, total, page, limit}` при той же таблице).

- **Индекс по `status` — отдельной миграцией.** Фильтр `?status=` уже есть, iter 5
  добавит пагинацию поверх — индекс завели сразу, второй миграцией
  (`20260719165211_ticket_status_index`). Побочный бонус: в истории миграций видно сам
  цикл «поменял схему → `migrate dev` → новый SQL-файл», а не только init-снимок.

## Грабли (Node повёл себя не как Python)

- **pnpm по умолчанию не даёт пакетам выполнять build-скрипты.** `postinstall` у prisma
  и `@prisma/engines` молча не выполнялся, пока их не внесли в `allowBuilds` в
  `pnpm-workspace.yaml`. В pip/uv мире такого барьера нет — setup-хуки пакетов
  выполняются без спроса; у pnpm это осознанная защита от supply-chain, и её надо
  явно разрешать по пакету.

- **Генерённый клиент — файлы в твоём репо, о которых надо договориться с тулингом.**
  В SQLAlchemy модели — просто твой код; здесь `src/generated/prisma/` надо исключить из
  git (`.gitignore`), а Biome и tsc натравлены на весь `src/` — сгенерённый код они
  проверяют наравне с рукописным (пока проходит; если начнёт мешать — исключим точечно).

- **Enum'ы Prisma — настоящие Postgres-типы.** Миграция делает `CREATE TYPE ... AS ENUM`,
  а не CHECK-констрейнт или строку — ровно то, ради чего в решении №5 семьи выкинут
  SQLite (у Prisma на нём enum'ов нет). Приятная сторона: недопустимое значение статуса
  отобьёт даже прямой SQL мимо API.

## Сознательные срезы и отложенные TODO

- **`GET /tickets` по-прежнему отдаёт голый массив** — маркер
  `dw-lite: голый массив → {items, total, page, limit}` в `routes/tickets.ts` стоит с
  iter 1, конверт списка придёт с пагинацией в iter 5 (контракт №5).
- **Полей `createdAt`/`updatedAt` нет.** Домен правила 3 — четыре поля, и он замораживается
  после iter 5; таймстемпы не нужны ни одной запланированной фиче, добавлять «на всякий
  случай» не стали.
- **Сервис `api` в compose не добавлен** — сейчас там только `db`; полная оркестрация с
  nginx и статикой фронта — iter 6 (контракт №6).
- **CI не знает про Postgres** — тесты в CI пойдут через postgres service container в
  iter 7; до тех пор прогон тестов требует локального `make db-up`.
