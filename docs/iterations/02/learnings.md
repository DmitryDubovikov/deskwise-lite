# Learnings 02 — Plugin system: дерево плагинов + `buildApp(deps)`

## Решения и почему

- **Store регистрируется внутри анонимного tickets-скоупа, а не в корне.** Можно было
  проще: `fp`-плагин в корень — и store виден всем. Но тогда демонстрация инкапсуляции
  (учебная цель итерации) выродилась бы в «глобальная переменная с лишними шагами».
  Анонимный плагин `async (tickets) => { ... }` в `app.ts` существует ровно как граница
  видимости: внутри — store и роуты-сиблинги, снаружи — `hasDecorator === false`. Когда в
  iter 4 придёт Prisma-клиент, у него будет готовый шов: тот же паттерн, другой плагин.

- **Зависимость едет в плагин через options, а не через замыкание на `deps`.**
  `ticketStorePlugin` получает store аргументом регистрации
  (`register(ticketStorePlugin, { store: ... })`), а дефолт `new Map()` подставляет
  composition root, не плагин. Так плагин остаётся переиспользуемой деталью без знания о
  том, откуда берутся зависимости, — все решения о составе живут в одном месте,
  `buildApp(deps)` (правило 6).

- **`deps` — интерфейс с optional-полями, а не позиционные аргументы.** `AppDeps` сейчас
  держит одно поле `ticketStore?`, в iter 4 добавится Prisma-клиент, в iter 8 —
  OpenAI-клиент; вызовы `buildApp()` в проде и `buildApp({ ticketStore: fake })` в тестах
  при этом не меняются. Питоний аналог — `create_app(settings=None, **overrides)`.

- **Health-тест переехал из `server.test.ts` в `app.test.ts` вместе со смертью
  `buildServer()`.** Тестовый файл следует за модулем, который проверяет; заодно в
  `app.test.ts` живут тесты формы приложения (инкапсуляция, спека без listen), а в
  `tickets.test.ts` — поведение роутов. Разделение по объекту теста, не по типу.

## Грабли («где Node повёл себя не как Python»)

- **Инкапсуляция — это область видимости, которой в Python-вебе просто нет.** Главный
  сдвиг мышления итерации: в FastAPI `include_router` глобален по эффекту — всё, что
  объявлено, видно отовсюду; здесь каждый `register()` порождает дочерний контекст, и
  декоратор из соседней ветки **не существует**. Это не защита от дурака, а управление
  зависимостями: видимость объявляется структурой дерева, а не дисциплиной импортов.

- **`fastify-plugin` хоистит ровно на один уровень — не в корень.** Интуиция «fp делает
  плагин глобальным» неверна: fp снимает инкапсуляцию только с самого плагина, и его
  декораторы ложатся в контекст, **где его зарегистрировали**. Зарегистрирован в корне
  (как `@fastify/swagger`) — виден всем; зарегистрирован в tickets-скоупе — виден только
  скоупу. Наш тест инкапсуляции держится ровно за эту разницу.

- **TypeScript про инкапсуляцию врёт — module augmentation глобальна.**
  `declare module "fastify"` докидывает `ticketStore` в тип `FastifyInstance` **целиком**:
  компилятор разрешит написать `app.ticketStore` и в корне, где в рантайме поля нет
  (`hasDecorator === false`, чтение даст `undefined`). Выразить «поле есть только внутри
  скоупа» система типов Fastify не умеет — это известное ограничение. Отсюда правило:
  границу видимости охраняет рантайм-тест (`app.test.ts`), а не tsc; после iter 4 store
  станет единственным, и augmentation перестанет быть натяжкой.

- **`register()` асинхронный, и порядок регистраций — это `await`-ы.** В Python порядок
  `include_router` безразличен; здесь swagger обязан встать до роутов (грабля ещё iter 1),
  а store — до роутов-сиблингов, и это выражается последовательными `await
  tickets.register(...)`. Дерево плагинов — код с порядком исполнения, а не декларация.

## Сознательные срезы (`dw-lite:`-маркеры в коде)

- `plugins/ticket-store.ts:4` — **`Map` вместо БД** остаётся:
  `// dw-lite: in-memory Map → Prisma-клиент (iter 4)`. Итерация меняла форму, не хранилище.
- `routes/tickets.ts:30` и `:48` — маркеры iter 1 переехали вместе с роутами без изменений
  (голый массив → контракт №5 в iter 5; плоский 404 → envelope №2 в iter 3).
- **Config-модуля по-прежнему нет** — порт 3000 захардкожен в `index.ts`; env не читается
  вовсе, поэтому и модулю пока нечего делать (появится с iter 3–4, спека: вне scope).
- **`onClose` для store не нужен** — у `Map` закрывать нечего; graceful shutdown приедет
  вместе с Prisma-клиентом в iter 4.
